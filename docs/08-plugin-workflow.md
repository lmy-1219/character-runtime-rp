# 第七部分 · 插件工作流：让自动化扛记忆

> 让自动化插件扛「记忆维护」，人只管审核和分发。费时间的是前期把提示词调通；跑通之后，几乎不用手工记东西。

整体流程：日常 RP → STMB 抓计划／事实、新建条目进仓库世界书 → 人工审核、分发到对应世界书 → World Info Recommender 当状态刷新机，扫上下文更新指定条目 → 接着 RP。全程开着「总结」当保险。

这套工作流里真正值得拆的，不是插件清单，是反复出现的两条原则——它们决定了为什么这么设计：**只记事实、不记关系／情感**；**讨论 ≠ 执行**。

### 一、滚动总结：只记事，不记情

这是个总结插件，一条极明确的边界：只记「事」，不记「情」。它的提示词开头是这么定的：

> 你是一个负责维护长篇角色扮演叙事连续性的事实便签助手。请阅读以下对话，在已有先前总结的基础上补充新发生的事实，生成一段简洁、信息密度高的中文滚动总结。

它记：时间地点锚点、关键剧情推进、重要对白要点（简短转述不照抄）、已做的决定／共识／承诺／未解决的分歧、可能被回调的具体元素（次要角色、物品、资产、价格、型号、数量）、还没解决还要回收的待办。它略：寒暄、重复确认、无信息量的过场、被后续推翻的内容、纯氛围描写。它严禁：写「关系升温」「感情加深」「更加信任」「产生羁绊」这类关系性总结；替角色判断感情状态；把行为解释成情绪或动机；用 2077 原作设定覆盖当前 AU。

为什么死活不让它总结情感：关系／情绪／亲密度这种判断是高维、模糊、还随模型不同而变的。你让插件去总结「关系升温」，不同模型可能得出完全不同的结论，而这些结论会反过来污染上下文的概率分布——等于让一个工具替模型把「他该有什么感受」提前拍死了。所以只记客观事实，让模型每轮基于事实自己「算」感受，而不是让插件下结论。

### 二、STMB：事实抽取三件套

STMB 从上下文抽三类信息，分别生成世界书条目：

1. **抓计划**：只抓「刚被确立、之前没记录过」的。什么算计划——有目标、至少一个具体步骤／方案／对象、由角色主动提出，不是随口一愿。最关键一条判断规则：讨论、同意、意图，都不等于执行；「讨论过」不算「做了」，「计划了」不算「完成了」。为啥这么严——条目的价值就在于区分「说过要做」和「真的做了」，AI 要是把讨论当执行，整个账本就废了。
2. **抓能力机制**：抓某个角色「新暴露／新澄清」的能力细节（怎么运作、能干什么、什么触发、代价、不能干什么），整合成一条、绝不打碎。战斗或对话里常会暴露之前没说清的能力，这些不落地，下一轮模型就忘了。
3. **提取新增认知**：抓「某条信息从 A 传到了 B」这个事实（说了／展示了／坦白了／共同活过来了）。设定里为真但没披露的不算。为啥要这个——V 知道的、强尼知道的、模型知道的，三者不一定一样。这插件保证认知边界不糊：模型不替 V「知道」V 没说过的事，也不让强尼忘记 V 已经告诉过他的事。

### 三、World Info Recommender（WREC）：只认证据的状态审稿员

WREC 是整套工作流的核心阀门：读上下文，判断指定条目的状态字段该不该更新。管两摊：计划状态更新、资产账本更新。它的心智是一句话——只认证据的法官。

你说「准备去做」，它不认为「做了」；「可能走这条路」，它不认为「确认了」；「很担心这件事」，它不认为「有进展」。角色的态度、情绪、担忧、亲密、信任、关系变化，统统不作为任务进展的证据。只有上下文明确写了「做了」，它才更新。

账本更比计划更严，连「估算」都不做：上下文没明说金额，不动；不估价、不脑补隐藏成本；没明说到账就不当它到账；没明说买了／收了就不当角色拥有。V 说「这委托给 3000 欧」，但钱真到账之前，WREC 不会把它写进现金里。它只记明确发生的交易，不记承诺。

WREC 和 STMB 抓计划那条，本质是同一条「讨论 ≠ 执行」在两个环节各设一道闸——一道在抽取（STMB 不把口头意图建成计划），一道在更新（WREC 不把口头意图翻成进度）。这是整个账本可信的根。

### 四、后续规划（有想法，提示词还没搞好）

现在 WREC 覆盖「计划状态」和「资产账本」。下一步想加「角色身体状态」模块（疲劳度、受伤程度、义体状态），把自动状态管理再扩一块。
