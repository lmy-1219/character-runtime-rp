# 第四部分 · 支柱三：分层记忆（世界书）

> 上下文窗口再大也装不下百万字。解法不是塞更多，是分层 + 按需加载——平时只让角色「底色」常驻，复杂信息用关键词和递归索引在需要时整条浮出、不需要时一个 token 不占。

1M token 看着多，但你把两百轮原文、设定、世界观、计划清单全塞进去，照样爆。就算不爆，信息密度一低，注意力就散——这其实是[第一部分](02-ai-thinking.md)「塞太满」那个问题在长程上的版本。何况平时根本拉不到 1M，我体感 20–32k 就够用。

我把世界书拆成五层，各管各的、更新频率不同：

1. **全局世界书**：世界观、势力、地图、系统规则。几乎不动。加载：常驻（AU 私设锁定）+ 关键词。
2. **角色设定世界书**：能力边界、对其他角色的态度、既定前史。几乎不动。加载：常驻 + 关键词。
3. **认知边界世界书**：角色对 user 已知的信息、共同经历。低频、只增不删。加载：关键词。
4. **阶段世界书**：当前计划、当前状态、当前相处模式。高频。加载：相处模式常驻；计划/状态用关键词高频联动。
5. **插件仓库世界书**：临时抓取的事实，待人工审核分发。不启用，只做中转站。

我那本阶段世界书现在的条目概览，直接贴出来看：

```
基调/与V相处   规则/计划推进   私有/Johnny-涟漪池   索引/水晶宫线
计划/对外叙事补充   计划/暗杀赫尔曼   计划/水晶宫复出   已确认资产
计划/蓝眼睛先生委托   计划/原型武器与生物覆写
```

命名规律是 `类别/具体`，类别就那么几种（基调／规则／私有／索引／计划／资产）——这套命名本身就是给我自己和插件看的路标。

条目怎么配，是这一层的重点，三种加载方式：

- **常驻**：永远在上下文，保证底色不丢（当前相处模式、特殊私设）。
- **关键词触发**：上下文冒出特定名词（「荒坂」「克里」「神舆」）才把相关背景拉进来，省得无关信息占位。
- **递归索引**：复杂任务链用一个「索引条目」当网关——日常 RP 它不触发，剧情一进这条任务线，索引被拉出来，它内部写的条目名再去触发具体细节。两级触发，复杂信息需要时整条浮出、不需要时一个 token 不占。

递归逻辑记一句口诀：**看谁是谁的爹**。儿子们和爷爷都勾「防止进一步递归」，当爹又当儿子的中间层可被递归、也可往下递归。简单粗暴但有效。

为什么要这么绕——还是回到注意力。一条复杂任务链（七个角色、三个目标、五条支线）如果全常驻，等于把一大坨低相关信息天天垫在概率底下，稀释一切。递归索引是把「相关性」做成开关：不相关时它彻底消失（零 token），相关时它整条、按依赖顺序浮出来。你不是在压缩信息，是在控制信息「什么时候存在」。

实测：做高复杂度任务时，完整触发索引 + 背景 + 目的 + 任务线 + Johnny 对相关角色的态度，全部世界书加起来大概吃 8000+ token（含所有书的八个常驻条目）。对 32k 上下文来说，还在黄金区间。

关键词和递归具体怎么设（我用纯关键词触发、不上向量化，低级但省钱）：

- 越靠底层的大概念（角色／物品／地点这种背景名词）：关键词设宽、内容写简单精准（够模型不犯常识错误就行）、设为可递归。
- 会影响角色表现的内容（领域反应机制、已发生事实）：主关键词用相关名词，再配一个可选过滤器防误触发（主关键词用名词、过滤器配相关动词）。
- 一整条计划链：同路径任务线合成一整条；单独建一个索引条目（只放各支线和背景的条目名、不放实际内容，保证日常 RP 零触发）。
